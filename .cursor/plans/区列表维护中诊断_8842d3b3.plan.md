# 区列表显示「维护中」诊断

## 结论：**不是客户端解析 bug，是服务端下发的区服状态**

在 [`parseMsgHeader` 修复](sdk/net/ClientMsgHandler.cpp) 后，区列表能正常解析。此时 UI 显示「维护中」表示客户端**如实展示**服务端 `Msg_S2C_ZoneEntryWire` 中的状态字段，属于预期行为。

客户端文案来源：[`ui/ServerListPanel.cpp`](ui/ServerListPanel.cpp) 将 `ZoneLoadStatus::Maintenance` 映射为 **「维护中」**（灰显、不可选）。

```mermaid
flowchart LR
    LS[LoginServer] -->|"S2C_ZONE_LIST_RSP"| Client[ClientMsgHandler]
    Client -->|"enabled + loadLevel"| Map[loadStatusFromWire]
    Map --> UI[ServerListPanel 维护中]
```

## 客户端何时判定为「维护中」

[`sdk/net/ClientMsgHandler.cpp`](sdk/net/ClientMsgHandler.cpp) 中 `loadStatusFromWire`：

| 条件 | UI 状态 |
|------|---------|
| `enabled == 0` | 维护中 |
| `enabled == 1` 且 `loadLevel == 3` | 维护中 |
| `enabled == 1` 且 `loadLevel == 0/1/2` | 畅通 / 繁忙 / 爆满 |

[`ui/ServerListPanel.cpp`](ui/ServerListPanel.cpp) 中 `zoneRowSelectable` 要求 `enabled && loadStatus != Maintenance`，因此维护区**无法点选确定**。

## 服务端何时下发 loadLevel=3

LoginServer 在 [`RPG_Server/LoginServer/ZoneInfoStore.cpp`](../RPG_Server/LoginServer/ZoneInfoStore.cpp) 的 `computeLoadLevel` 中计算：

1. **`serverlist.xml` 中 `enabled="0"`** → 直接维护
2. **Super 运行时上报缺失或超过 30s 未更新**（`ZONE_RUNTIME_STALE_MS`）→ 维护（LoginServer 启动后 60s 宽限期内可能仍显示畅通）
3. **`runtime.alive == false` 或 `gatewayCount == 0`** → 维护

区列表组装见 [`RPG_Server/LoginServer/LoginAuthService.cpp`](../RPG_Server/LoginServer/LoginAuthService.cpp)：始终发 v2 条目（112B），`enabled` 来自 xml，`loadLevel` 由 `fillGatewayLoadFields` 实时计算。

当前服务端配置 [`RPG_Server/LoginServer/serverlist.xml`](../RPG_Server/LoginServer/serverlist.xml) 中 `RPG一区` 为 `enabled="1"`，因此**更常见原因是运行时未就绪**（只开了 LoginServer，Super/Gateway 未启动或未注册），而非 xml 手动关区。

## 与「区服维护中」登录错误的区别

若是在**登录阶段**看到「区服维护中」/「区服不可用」，来自 LoginServer 登录校验（[`LoginAuthService.cpp` L255-271`](../RPG_Server/LoginServer/LoginAuthService.cpp)），同样是服务端拒绝，不是客户端误判。

## 排查步骤（优先查服务端）

1. **确认区列表已成功拉取**  
   客户端日志应有 `ZoneListSession：收到 N 个区服`（[`net/ZoneListSession.cpp`](net/ZoneListSession.cpp)），而不是「格式错误」类文案。

2. **查看 LoginServer 调试日志**  
   搜索 `区列表判定:` 一行，关注：
   - `enabled`（应为 1）
   - `gateway`（应为 >0）
   - `runtimeAlive`（应为 1）
   - `loadLevel`（3 = 维护）

3. **启动完整服务链**（开发环境典型缺项）  
   - LoginServer（9010）
   - SuperServer（serverlist 中 `superPort=9000`）
   - Gateway（向 LoginServer 注册）  
   仅 LoginServer 在线时，多数情况下 `gatewayCount=0` 或 runtime 过期 → `loadLevel=3` → 客户端显示维护中。

4. **核对 serverlist.xml**  
   确认目标区 `enabled="1"`；改后需重启 LoginServer。

5. **等待宽限期**  
   LoginServer 刚启动 60s 内，无 runtime 时可能暂时显示「畅通」；超过宽限期且无 Super 心跳后会变为「维护中」。

## 是否需要改客户端代码？

**当前不需要**。客户端逻辑与 [`README.md`](README.md) 说明一致：v2 协议下按 `loadLevel` 显示真实负载/维护状态。

若希望改善联调体验，可选增强（非必须）：

- 在 `parseZoneListRsp` 成功时打一条 `info` 日志：`zoneId / enabled / loadLevel / gatewayCount`，便于对照 LoginServer 日志
- UI 在 `loadLevel==3 && enabled==1` 时显示更具体提示（如「网关未就绪」），需服务端补充原因字段或客户端根据 `gatewayCount==0` 推断

## 预期修复后的联调结果

Super + Gateway 正常注册后，LoginServer 日志 `loadLevel=0`，客户端区列表应显示 **「畅通」** 且可选中进入登录流程。
